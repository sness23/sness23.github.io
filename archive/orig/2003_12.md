<p>ARP/wARP errors</p>

<p>I got the following error in ARP/wARP:</p>

<pre>
cp: cannot stat `/home/sness/Paper/arp-warp-all/arp_warp_runs/haptbrSHARP/1_mir_model_1.pdb': No such file or directory
grep: arp_0.pdb: No such file or directory
grep: arp_0.pdb: No such file or directory
 Refmac_5.1.24:  No Scatterers

<p></pre></p>

<p>This was due to an error in the ARP/wARP scripts where MAPMASK was making the wrong sized maps.  You can fix this by changing the <b>xyzlim</b> keyword in the ARP/wARP .par (parameter) file.</p>

<h3 class="title"></h3>

<p>If you get the following error when running Solomon:</p>

<pre>
 Non Primitive Spacegroup of order   2
 FCS Multiplied by   2
                                                                                                                                                               
 Check map header agrees with fixed requirements for SFcalc for this  spacegroup.
 Check Nxyz     180 196 204    180 196 204
                                                                                                                                                               
 Check map header agrees with fixed requirements for SFcalc for this  spacegroup.
 Check Iuvw       0   0   0      3   1   2
 SFALL:   **** Fatal disagreement between input info and map header
Times: User:       0.1s System:    0.0s Elapsed:     0:00
>>>>>> CCP4 library signal ccp4_general:Cannot find input file (Success)
     raised in ccp4setenv <<<<<<
</pre>

<p>It's because your map has the slow, intermediate, fast sections in the wrong order.  One way to fix this is to use SFTOOLS to read your map in and output it in the correct order:</p>

<pre>
sftools
mapin input.map
mapout output.map order c a b
</pre>

<h3 class="title"></h3>

<p>Ok, here's a very useful little change to make to the xemacs list-mode.el source file:</p>

<pre>
;              (frame-width (or (last-nonminibuf-frame)
;                   (selected-frame)))
; sness
               (window-width (get-lru-window (last-nonminibuf-frame)))
</pre>

<p>Comment out the first two lines and add the last line.  This solves a very annoying bug when using Xemacs in tty mode with a C-x 3 vertically split screen.  The bug is that when you do completion in this split window, the contents are created before the window is created, and if the window isn't as wide as the frame, part of the window will be truncated.  This hides some of the filenames that you're looking for, and is quite irritating.  </p>

<p>I found this great patch on the web <a href="http://list-archive.xemacs.org/xemacs-beta/200009/msg00137.html">here</a>.  It looks like it might be going into the beta xemacs 21.5. </p>

<p>Hopefully all of this verbosity will help google find this page for you, dear reader, with a variety of search terms.  Bon chance. </p>
